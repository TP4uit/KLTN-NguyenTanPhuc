# Demo Runbook

This runbook describes the current local MVP flows for the browser demo. It assumes a local Hardhat node, MetaMask connected to `localhost:8545`, and frontend assets synchronized from the generated proof/registry fixtures.

The flows are for thesis demonstration only. They are not production identity management, production voter registration, or production election infrastructure.

## State Boundaries

- On-chain contract state: `Election` address, lifecycle state, Merkle root, nullifier usage, and candidate vote counts. Resetting browser state does not change any of this.
- Frontend-local demo state: demo auth session/users, voter registrations, local identity secrets, UI vote-session cache, and preview/export state stored in browser storage.
- Generated proof/registry fixtures: files generated by root scripts, including `registry.local.json`, `vote.calldata.local.json`, browser proving assets, and deployment metadata.

Public approved `identityCommitment` values may appear only as public registry commitments in registration evidence and registry preview exports. Do not export identity secrets, passwords, raw proofs, raw nullifiers, vote choices, candidate choices, transaction hashes, wallet addresses, private keys, or private wallet data.

## Start the Local Stack

Terminal 1:

```shell
npm run node:local
```

Terminal 2:

```shell
npm run registry:generate
npm run proof:generate
npm run proof:calldata
npm run frontend:sync-fixtures
$env:LOCAL_ELECTION_AUTO_OPEN='false'; npm run deploy:localhost; Remove-Item Env:LOCAL_ELECTION_AUTO_OPEN
```

Terminal 3:

```shell
cd frontend
npm install
npm run dev
```

In MetaMask, use:

- RPC URL: `http://127.0.0.1:8545`
- Chain ID: `31337`
- Currency symbol: `ETH`

Import a funded Hardhat account from the node output if needed. Use the deployer/admin account for `/admin` lifecycle actions.

## Static Fixture Mode Demo

Use this when you want the original fixture-compatible path.

1. Sign in with the admin demo account.
2. Open `/admin` and connect MetaMask with the deployer/admin account.
3. In the Admin Demo Mode Guide, choose Static Fixture Mode. This fills the static fixture Merkle root only; it does not submit a transaction by itself.
4. Confirm `setMerkleRoot` in the Admin page.
5. Confirm `openElection`.
6. Sign in with the seeded voter demo account or another fixture-compatible approved voter.
7. Open `/dashboard`.
8. Cast a vote using the static/fixture-compatible submit path.
9. Open `/results` and refresh on-chain tallies.
10. Export Results audit JSON or the public evidence package.
11. Open `/audit` as an auditor or admin and validate the export.

Expected root behavior:

- `demoMode` should be `STATIC_FIXTURE`.
- `staticFixtureRootMatchesResultsRoot` should be true in evidence package checks.
- Dynamic Poseidon submit may be blocked because the contract root is not the dynamic preview root.

## Dynamic Poseidon Mode Demo

Use this when you want approved local Poseidon commitments to define the preview root.

1. Sign in as a voter and create a local voter registration from `/dashboard` if needed.
2. Sign in as admin and open `/admin`.
3. Review pending registrations. Approve registrations intended for the demo.
4. Confirm that Registry Preview includes approved `POSEIDON` and `FIXTURE_POSEIDON` commitments as compatible leaves.
5. Confirm that old or missing-scheme `SHA256_DEMO` registrations are listed as incompatible, not included in the Poseidon preview leaves.
6. In the Admin Demo Mode Guide, choose Dynamic Poseidon Mode. This fills the dynamic preview root only; it does not submit a transaction by itself.
7. Confirm `setMerkleRoot`.
8. Confirm `openElection`.
9. Sign in as an approved Poseidon voter and open `/dashboard`.
10. Check Dynamic Vote Readiness. Dynamic submit requires the Open election state, a contract root equal to the dynamic preview root, local identity material, proof input preview readiness, and no current UI-session vote.
11. Cast a vote using guarded Dynamic submit.
12. Open `/results` and refresh on-chain tallies.
13. Export Results audit JSON or the public evidence package.
14. Open `/audit` as an auditor or admin and validate the export.

Expected root behavior:

- `demoMode` should be `DYNAMIC_POSEIDON`.
- `registryPreviewRootMatchesResultsRoot` should be true in evidence package checks.
- Static fixture submit may be blocked because the contract root is not the fixture root.

## Results, Audit, and Evidence Package Review

Use `/results` after connecting MetaMask to read current contract state.

Results audit JSON includes public contract/root context:

- candidate tallies
- total votes
- lifecycle state
- latest block
- Merkle root
- demo mode attribution
- static/dynamic root match flags

The public evidence package combines:

- Results audit JSON
- frontend-local registration evidence summary
- Poseidon registry preview
- demo mode / Merkle root attribution
- safety warnings
- verification checks

Use `/audit` to import either raw Results audit JSON or the full public evidence package. For a package import, review:

- Evidence Package Review metadata
- On-chain Results Audit card
- Registration Evidence card
- Registry Preview card
- Root Alignment card
- Final Auditor Verdict
- warnings and limitations

The normalized review report is also public. It includes package metadata, verdict, checks, warnings, errors, and live comparison rows if you ran live comparison. It must not include secrets, raw proofs, raw nullifiers, vote choices, transaction hashes, wallet/private data, or private keys.

Evidence package review is public demo audit support only. It is not a cryptographic proof of per-vote provenance.

## Resetting Local Browser State

The Admin Demo Runbook includes reset controls for frontend-local browser state.

Reset controls can clear:

- voter registrations
- local identity secrets
- local demo auth session/users
- registered UI cache keys

Reset controls do not reset:

- contract Merkle root
- election lifecycle state
- on-chain votes
- deployed contract addresses
- generated `registry.local.json`
- generated `election.local.json`
- generated `vote.calldata.local.json`
- browser proving assets

To reset contract state, restart the local Hardhat node and redeploy. To reset generated fixtures, rerun the root fixture/proof scripts.
